# Ever Club Members App

**Current Version**: 8.95.6 (March 21, 2026)

## Overview
The Ever Club Members App is a private members club application designed for golf and wellness centers. Its primary purpose is to serve as a central digital hub for managing golf simulator bookings, wellness service appointments, and club events. The project aims to enhance member satisfaction and operational efficiency through comprehensive membership management, facility booking, and community-building tools, ultimately creating a seamless digital experience for club members and staff.

## System Architecture

### Core Architecture
- **Naming Conventions**: `snake_case` for PostgreSQL; `camelCase` for Drizzle schemas, API JSON, and React/TypeScript frontend. Raw database rows are not exposed in API responses.
- **Type Safety**: Strict TypeScript with zero errors; no `as any`. All raw SQL results are typed.
- **Database Interaction**: Drizzle ORM query builders or parameterized `sql` template literals are mandatory.
- **Timezone**: All date/time operations explicitly use Pacific Time (`America/Los_Angeles`).
- **Audit Logging**: All staff actions are logged.
- **API/Frontend Consistency**: API response field names align with frontend TypeScript interfaces.
- **Database Migrations**: The project uses `drizzle-kit push` for schema sync in development. **CRITICAL: `npm run db:push` now runs `scripts/safe-db-push.ts`** which wraps `drizzle-kit push --force` with pre/post column snapshot comparison. It blocks column drops on protected tables (users: google_id, google_email, apple_id, stripe_customer_id, etc.) and attempts emergency restoration if a drop is detected. The raw `drizzle-kit push` is available as `npm run db:push:unsafe` but should NEVER be used directly. **Root cause of March 17 2026 data wipe**: Multiple task agents ran `drizzle-kit push --force` from subrepls that had older schemas missing google auth columns, causing Drizzle to DROP+re-ADD the columns and wiping all 720 users' Google linking data. For deployment, Replit's platform runs Drizzle file-based migrations from the `drizzle/` folder. The `drizzle.__drizzle_migrations` tracking table must have entries for all applied migrations — if it's empty, deployment will fail trying to re-apply all migrations from scratch. The build step runs `tsx scripts/sync-migration-tracking.ts` which automatically bootstraps the `drizzle` schema and `__drizzle_migrations` table if missing, reads `drizzle/meta/_journal.json`, and registers all 33 journal entries (MD5 hash of each SQL file + `when` timestamp). The script is idempotent — safe to run multiple times. When adding new migrations, add them to `drizzle/meta/_journal.json`; the sync script will auto-register them on next build/deploy.
- **Database Migration History**: On March 18, 2026, a Supabase migration was attempted but reverted. Both dev and production now use the Replit-provisioned PostgreSQL (helium) database. The `DATABASE_URL` is runtime-managed by Replit — do not override it with secrets. A backup of the data is saved in `migration_backup/`.
- **Database & Data Integrity**: PostgreSQL, Supabase Realtime, and Drizzle ORM with CASCADE constraints. DB-level enforcement includes unique indexes on `stripe_customer_id`, `hubspot_id`, `google_id`, and `apple_id` (partial, non-null), CHECK constraint preventing `billing_provider='mindbody'` with an active `stripe_subscription_id`, guest pass arithmetic checks (`guest_passes_usage_check`), booking time validity checks, exclusion constraint preventing overlapping bookings, membership_status lowercase CHECK, `trg_cascade_user_email_delete` trigger cascading user deletes to email-linked tables, and `trg_validate_guest_pass_member` trigger preventing orphan guest passes. **DB-level state machines**: `trg_booking_status_machine` enforces valid booking status transitions (terminal statuses: attended, no_show, cancelled, declined, expired), `trg_membership_status_machine` enforces valid membership status transitions (terminal: archived, merged). Both use `current_setting('app.bypass_status_check', true)` escape hatch for data migration/repair. **DB-level guards**: `trg_guard_stale_pending` prevents approving bookings >2hr past start, `trg_guard_attended_unpaid` blocks attended transition with unpaid fees, `trg_prevent_archived_*` (5 tables) blocks inserts for archived members, `trg_cleanup_fee_on_terminal` auto-cancels fee snapshots on terminal booking status, `trg_auto_expire_stale_tours` marks stale tours as no_show. **Dedup indexes**: `idx_users_email_stripe_unique` (one Stripe customer per email), `idx_bookings_invoice_unique` (one active invoice per booking). 23 total integrity checks: 8 external-system checks run on scheduled runs (Stripe subscription sync/billing orphans/orphaned subscriptions/tier reconciliation, HubSpot, Trackman, MindBody x2), 15 DB-enforced/internal checks run on manual trigger only. Items Needing Review removed (workflow queue). Duplicate Stripe, Orphaned Payment Intents, Invoice-Booking, Guest Pass Drift, Stale Pending Bookings, Archived Member downgraded to medium since DB now prevents new occurrences. Email normalization, status case, billing provider, staff role sync, and participant user_id linking are handled by DB triggers — retired from periodic auto-fix. **Runtime FK constraints** (on `booking_requests.session_id`, `booking_requests.closure_id`, `booking_participants.user_id/session_id/guest_id`, `booking_wallet_passes.booking_id/member_id`, `booking_fee_snapshots.booking_id/session_id`, `guest_pass_holds.booking_id`, `conference_prepayments.booking_id`, `wellness_enrollments.class_id`) are created ONLY in production by `db-init.ts` — never in dev — to prevent Replit's deployment system from detecting dev/prod DB diffs and generating failing migrations. All use `NOT VALID` + separate `VALIDATE CONSTRAINT` with orphan cleanup.
- **Real-time Updates**: WebSocket broadcasting for booking and invoice changes via Supabase Realtime. React Query with WebSocket-driven cache invalidation is used for the Staff Command Center.
- **Member Dashboard**: Features a chronological card layout for bookings, events, and wellness sessions with "Add to Calendar" functionality. **Dashboard API split (v8.87.36)**: The monolithic `/api/member/dashboard-data` endpoint was replaced by 8 independent endpoints under `/api/member/dashboard/` — `bookings`, `booking-requests`, `rsvps`, `wellness`, `events`, `conference-rooms`, `stats`, `announcements`. Each endpoint is fetched in parallel with independent error handling (one section failing doesn't break others). Frontend uses React Query with key pattern `['member', 'dashboard', email, section]`; `refetchAllData()` invalidates the `['member', 'dashboard']` prefix (covers all 8). The legacy monolithic endpoint is kept for backward compatibility but has no frontend consumers.

### Core Domain Features
- **Booking & Scheduling**: "Request & Hold" model, unified participant management, calendar synchronization, auto-complete scheduler, and conflict resolution. Integrates with Trackman webhooks and handles guest/overage fees. Staff golf lessons create only `availability_blocks` (no `facility_closures` notice records). The Google Calendar sync (`syncInternalCalendarToClosures`) explicitly skips lesson-titled events to prevent them from creating notices. Trackman webhook auto-approval transfers all booking participants to the new session. Staff can assign additional players to sessions. Booking approvals are separated from session creation for cleaner code paths. Booking cancellation uses a transactional DB status change with post-commit best-effort side effects (guest pass refund, usage ledger cleanup, Stripe refund, calendar deletion). **Trackman imports are usage-tracking only — all trackman-sourced participants get `payment_status = 'waived'` and the balance endpoint excludes `trackman_import`/`trackman_webhook` sessions from fee calculations.** V2 Trackman webhook auto-link (`tryMatchByBayDateTime`) uses a tight 5-minute tolerance with end-time verification and refuses ambiguous matches (multiple candidates). CSV import corrects session owner mismatches when the authoritative customer data differs from the auto-linked owner. **Booking owner invariant**: all 9 paths that set `booking_requests.user_email` also update `booking_requests.user_id` and the `booking_participants` session owner row — including CSV unmatched→matched, CSV placeholder merge, CSV ghost update, staff assign/change-owner, staff link-trackman-to-member, and admin resolution. **Participant overlap protection**: Booking creation checks both the owner AND each member participant for time conflicts (owned or participant bookings at overlapping times). Conference room frontend uses `.filter()` (not `.find()`) to check ALL simulator bookings when blocking overlapping time slots. `checkExistingBookings` and `checkExistingBookingsForStaff` both include participant bookings via subquery. During booking approval, conflicting participants are silently skipped (not added to the session) with a warning log — the approval itself is not blocked.
- **Fees & Billing**: Unified fee service, dynamic pricing, prepayment, and guest fees based on a "one invoice per booking" architecture. Supports dual payment paths and existing payments. Roster changes trigger fee recalculation; payment locks the roster. Staff can manage conference room prepayments on behalf of members. Fee recalculation skips already-paid participants. Draft invoices are cleaned up when bookings are cancelled. Invoices are voided on permanent booking deletion. **Usage ledger queries must filter out cancelled bookings**: `ledger_usage` CTE in `unifiedFeeService.ts` and `getDailyUsageFromLedger` in `usageCalculator.ts` use `EXISTS (... status NOT IN cancelled/declined/cancellation_pending/deleted)` to only count usage from sessions with at least one active booking. Conference room and simulator daily allowances are tracked separately (`daily_conf_room_minutes` vs `daily_sim_minutes`). **Invoice PI architecture**: Member portal interactive payments use `collection_method: 'charge_automatically'` with `expand: ['payment_intent', 'confirmation_secret']` on `finalizeInvoice()` to get the PI's `client_secret` synchronously. The member chooses their payment method via Stripe Payment Element. When the PI succeeds, Stripe auto-settles the invoice — no OOB needed. **Unified payment modal**: Both Dashboard and History page use `MemberPaymentModal` for booking-linked invoices (detected via `metadata.bookingId`). `InvoicePaymentModal` is only used for non-booking invoices. The fee summary shows per-participant line items with fee type descriptions (Overage Fee, Guest Fee) computed by `describeFee()` using `PRICING.OVERAGE_RATE_DOLLARS`. Staff paths (booking-fees.ts) keep `charge_automatically` via `finalizeAndPayInvoice` for explicit charge-saved-card, terminal, and mark-as-paid flows. OOB reconciliation (`paid_out_of_band`) is reserved exclusively for terminal/in-person payments (`card_present`/`interac_present`) and staff mark-as-paid. `buildInvoiceDescription` uses only the booking reference — fee details come from Stripe line items. Stale open invoices (amount mismatch with recalculated fees) are voided and recreated.
- **Member Lifecycle**: Membership tiers, QR/NFC check-in, and onboarding processes. QR scans route to booking check-in or walk-in.
- **Walk-In Visit Tracking**: Walk-in visits are recorded via QR/NFC scan, syncing to HubSpot and broadcasting WebSocket events.
- **Error Handling**: Prohibits empty catch blocks; all must re-throw, log, or use `safeDbOperation()`. `safeDbTransaction()` uses Drizzle's native `db.transaction()` with automatic rollback and staff alert notifications on failure (rewritten from raw `PoolClient` in v8.75.0). Global Express error middleware (`server/index.ts`) catches unhandled route errors and returns JSON 500 responses instead of raw HTML. Custom `AppError` class (`server/core/errors.ts`) replaces plain thrown objects with proper Error instances preserving stack traces. `sql.raw()` calls validated against column allowlists to prevent SQL injection. Session/billing lookups re-throw DB errors instead of silently returning null/empty. **Error message extraction rule**: All `catch` blocks in server code must use `getErrorMessage(err)` from `server/utils/errorUtils.ts` — never `(err as Error).message` (unsafe cast crashes on non-Error objects).
- **Authentication**: All mutating API routes require authentication. Sign-in methods: OTP (email verification code), Google OAuth, and Apple Sign-In (`server/routes/auth-apple.ts`). Apple auth uses `jose` library for JWKS token verification with Apple's public JWKS endpoint. User matching: Apple ID → Apple email → resolved primary email → direct email. Auto-links Apple account on first successful login. Both Google and Apple auth use explicit role allowlists (`'admin' | 'staff' | 'member'`) — unknown/corrupt role values default to `'member'`, preventing inactive users from bypassing membership gates. Profile Connected Accounts section supports link/unlink for both Google and Apple accounts (`/api/auth/google/status|link|unlink`, `/api/auth/apple/status|link|unlink`). **Status mapping**: All auth paths (OTP, Google, Apple) use a `statusMap` to normalize DB `membership_status` to display status (e.g. `'past_due'` → `'Past Due'`). The `/api/auth/session` endpoint refreshes status from DB on every call. Frontend `AuthDataContext` passes real status through (never hardcodes `'Active'`). Active-access checks throughout the app use `ACTIVE_STATUSES: ['active', 'trialing', 'past_due']` — these statuses grant full member access. **Domain aliasing (v8.89.3)**: Staff email domain migrated from `evenhouse.club` → `everclub.co`. `getAlternateDomainEmail()` in `server/core/utils/emailNormalization.ts` maps between domains. `getStaffUserByEmail()` and `GET /api/staff-users/by-email/:email` check both domains using SQL `IN` clause. All placeholder email patterns, notification filters, and Trackman constants recognize both `@visitors.evenhouse.club` and `@visitors.everclub.co`. Primary domain for new records is `everclub.co`.
- **Apple Wallet Pass**: Members can add a digital membership card to Apple Wallet from their profile. Pass Type ID: `pass.club.everclub`, Team ID: `S3DL4A3ZND`. Configuration stored in DB settings (`apple_wallet.*`). Signing certificates stored as environment secrets (`APPLE_WALLET_CERT_PEM`, `APPLE_WALLET_KEY_PEM`) — must be PEM text format, not binary DER. Pass generation uses `node-forge` for PKCS#7 signing with bundled WWDR G4 intermediate cert. Route: `server/routes/walletPass.ts`, generator: `server/walletPass/passGenerator.ts`. `Content-Disposition` must be `inline` (not `attachment`) for iOS to recognize the pass.

- **Rate Limiting**: Public endpoints creating database records are rate-limited. Subscription creation endpoints have a dedicated `subscriptionCreationRateLimiter` and an in-memory per-email operation lock (v8.58.0). OTP verification uses three-tier rate limiting: per-IP+email (5 attempts), per-IP global (15 attempts), and per-email aggregate (20 attempts). Keys always include IP (fallback to 'unknown') to prevent unauthenticated lockout of legitimate users. Rate limiter key generators use `String()` coercion before `.toLowerCase()` to prevent TypeError crashes from non-string input (v8.69.0). The in-memory lock cleanup interval uses `.unref()` to avoid pinning the event loop (v8.69.0).

Existing incomplete subscriptions can be reused via payment intent refresh. Idempotency keys prevent duplicate Stripe charges. Frontend guardrails: email pre-check on blur (`GET /api/visitors/check-email`), form submission debounce refs, 5-second post-creation cooldown, session-level recent-creation alerts (10-min window), and "Recently Added (24h)" directory filter.
- **Scheduler Robustness**: All 25 scheduler files (28 logical schedulers — `integrityScheduler.ts` contains 3) have overlap protection — `isRunning` flags (19), named guards (3), `isProcessing`/`isSyncing` (2), recursive setTimeout (2). Also uses catch-up windows, claim slots, and persistent notification deduplication (6-hour windows for waiver review and stuck booking alerts). **Staggered startup**: `initSchedulers()` staggers all 28 logical schedulers across ~270s in 6 waves (10s apart) to prevent database connection spikes at deployment. Wave order: real-time → booking/calendar → notifications → financial → HubSpot/external → cleanup. **Instant DB triggers** (`trg_auto_billing_provider`, `trg_sync_staff_role`, `trg_link_participant_user_id`) handle data fixes at write time; schedulers are daily safety nets only. Linked email users should be merged into the primary via `userMerge.ts` — the system no longer copies tiers to linked user records (`trg_copy_tier_on_link` was removed). **Error reporting**: `alertOnScheduledTaskFailure` extracts `error.cause` from Drizzle ORM errors to surface the actual database error in staff alerts (Drizzle wraps PG errors and hides the real message in `cause`).

Manual Stripe refunds trigger full resource teardown (guest pass refund, usage_ledger cleanup). Customer linking requires authenticated metadata (`source=app` or `userId`). Prevents double-charging, ensures orphaned invoice cleanup, uses optimistic locking for booking status transitions, and maintains atomicity for critical operations. `usage_ledger` has `ON DELETE CASCADE`. `booking_fee_snapshots` has `ON DELETE CASCADE` on both `booking_id` and `session_id`. `booking_participants` has `ON DELETE CASCADE` on `session_id`. - **Critical RLS Fix (v8.88.6, March 19 2026)**: Discovered that Supabase had enabled Row-Level Security (RLS) with a `deny_all` policy on ALL 82 application tables. This silently blocked every SELECT/INSERT/UPDATE from the application's database role, causing `getTierLimits()` to return 0 daily minutes for all tiers. **Impact**: Every member was charged overage fees ($50/hr) because the fee calculator fell back to defaults when tier queries failed. Also caused Stripe product sync failures (products couldn't read membership_tiers) and Apple Wallet pass generation failures. **Fix**: Added RLS disable + `deny_all` policy cleanup as the first step in `ensureDatabaseConstraints()` in `db-init.ts`. Also added Stripe feature name update logic to `syncTierFeaturesToStripe()` to correct stale "/month" labels to "/year" for guest pass features.
